# 쿠팡 지출 정리 (가칭: CoupangLedger) — 전체 설계문서

> 이 문서는 Claude Code에게 그대로 전달하는 마스터 설계도다.
> 각 Phase를 순서대로 진행하고, Phase를 건너뛰지 않는다.
> 코드 작성 시 이 문서의 "기술 아키텍처"와 "코딩 규칙" 섹션을 항상 참조할 것.

---

## 0. 한 줄 정의

**쿠팡 주문내역을 클릭 한 번으로 추출해서, 자동 분류된 지출 리포트로 보여주는 크롬 확장.**

- 타깃: ① 가계부 쓰는 개인 ② 회사 비품을 쿠팡으로 사는 직장인/자영업자(경비 정산용)
- 검증된 수요: 기존 확장 "쿠팡 결제내역 추출" — 마케팅 없이 유저 1,000명, 평점 5.0, 2025-04 이후 업데이트 중단
- 우리의 차별점: 기존 도구는 CSV 저장이 끝. 우리는 **분류 + 리포트 + 반복 사용 이유**를 만든다.
- 마케팅 전제: 사람 대면 0. 크롬스토어 검색 노출 + SEO 블로그 글로만 유입.

## 1. 제품 원칙 (모든 결정의 기준)

1. **로컬 우선**: 주문 데이터는 사용자 브라우저(IndexedDB)에만 저장. 서버로 안 보냄. → 프라이버시가 곧 마케팅 문구.
2. **한 번의 클릭**: 모든 핵심 동작은 버튼 1개로 끝나야 한다. 설정 요구 최소화.
3. **깨져도 조용히 죽지 않기**: 쿠팡 화면 개편으로 파싱이 실패하면 사용자에게 명확히 알리고, 원격 설정으로 코드 배포 없이 복구 가능해야 한다.
4. **무료가 충분히 좋아야 한다**: 무료 기능만으로 기존 1,000명짜리 확장을 완전히 대체. 유료는 "더 편함"을 판다.
5. **작게 출시, 빨리 반복**: Phase 1은 3주 안에 스토어에 올라가야 한다. 완벽주의 금지.

## 2. 사용자 시나리오

### 시나리오 A — 가계부 유저 (무료 → 유료 전환 후보)
1. 크롬스토어에서 "쿠팡 엑셀" 검색 → 설치
2. 쿠팡 주문목록 페이지 접속 → 확장이 "내역 가져오기" 버튼 표시
3. 클릭 → 진행바 → "247건 수집 완료"
4. 대시보드에서 월별 지출, 카테고리별 파이차트 확인
5. CSV/엑셀 다운로드
6. (유료 유도 지점) "AI 자동 분류로 '식품/생활용품/육아…' 정확하게 나눠보세요"

### 시나리오 B — 경비 정산 직장인 (유료 전환율 최고 예상)
1. 매달 말 법인카드 정산 때문에 검색 → 설치
2. 기간 지정(3/1~3/31) 후 추출
3. 주문번호별 합산 금액으로 카드 명세서와 대조
4. (유료 유도 지점) "매달 자동으로 수집 + 구글시트 연동"

### 시나리오 C — 파싱 실패 상황 (신뢰 유지)
1. 쿠팡 화면 개편 → 수집 버튼 클릭 시 0건
2. 확장이 자동 감지: "쿠팡 화면이 변경된 것 같습니다. 수정 중입니다 (보통 48시간 내)"
3. 원격 설정(셀렉터 JSON) 업데이트 → 재설치 없이 자동 복구

## 4. 기술 아키텍처

### 4.1 스택
- **Manifest V3** 크롬 확장 (파이어폭스/엣지는 Phase 4에서 검토)
- 언어: **TypeScript**
- 번들러: **Vite + @crxjs/vite-plugin**
- UI: 대시보드는 확장 내 전체 페이지(chrome-extension://.../dashboard.html), **React + Recharts**. 팝업은 최소한의 상태 표시만.
- 저장소: **IndexedDB** (Dexie.js 래퍼) — chrome.storage는 용량 한계로 주문 데이터에 부적합. 설정값만 chrome.storage.sync.
- 백엔드(Phase 3부터만): **Cloudflare Workers** 1개 — ① 라이선스 검증 ② AI 분류 프록시. 그 외 서버 없음.

### 4.2 컴포넌트 구조
```
[content script]  쿠팡 주문목록 페이지에 주입
   ├─ 수집 버튼 UI 삽입
   ├─ DOM 파싱 (셀렉터는 selectors.json 기반, 하드코딩 금지)
   └─ 파싱 결과 → service worker로 메시지
[service worker]  (background)
   ├─ IndexedDB 저장 (Dexie)
   ├─ 원격 셀렉터 설정 fetch (24h 캐시)
   └─ (Pro) 주기적 수집 알람 (chrome.alarms)
[dashboard page]  확장 내부 페이지
   ├─ 리포트/차트/내보내기
   └─ 분류 편집 UI
[popup]  수집 상태 + 대시보드 열기 버튼만
```

### 4.3 데이터 모델
```typescript
interface OrderItem {
  id: string;            // orderId + productId 조합
  orderId: string;
  orderedAt: string;     // ISO 8601
  productName: string;
  quantity: number;
  unitPrice: number;     // 원
  totalPrice: number;    // 원
  status: string;        // 배송완료/취소 등 원문 그대로
  category: string | null;      // 분류 결과
  categorySource: 'rule' | 'ai' | 'user' | null;
  collectedAt: string;
  raw?: string;          // 파싱 원본 (디버그용, 30일 후 삭제)
}

interface CategoryRule {
  keyword: string;       // 상품명 포함 키워드
  category: string;
  priority: number;      // 사용자 수정 = 최우선
}
```
기본 대분류 8종: `식품 / 생활용품 / 뷰티·건강 / 패션 / 육아 / 가전·디지털 / 문구·취미 / 기타`

### 4.4 파싱 전략 (가장 중요한 섹션)
1. **셀렉터 외부화**: 모든 DOM 셀렉터는 `selectors.json`에 정의. 코드에는 셀렉터 문자열이 한 줄도 없어야 한다.
2. **원격 갱신**: service worker가 GitHub raw URL(공개 repo)의 `selectors.json`을 24시간마다 fetch → 로컬 캐시. 쿠팡 개편 시 repo에 JSON만 커밋하면 전 유저 복구. (MV3에서 원격 '코드'는 금지지만 원격 '데이터'는 허용)
3. **점진 파싱 + 검증**: 각 필드 파싱 후 유효성 검사(금액이 숫자인지, 날짜 형식인지). 필드 단위 실패는 raw 보존하고 계속, 전체 실패율 30% 초과 시 중단 후 시나리오 C 메시지.
4. **예의 있는 순회**: 페이지 이동 간 1.5~2.5초 랜덤 대기. 한 번에 최대 100페이지. 병렬 요청 금지. (봇 감지 회피가 아니라 서버 부담 최소화 관점 — 사용자가 직접 넘겨보는 속도 수준 유지)
5. **⚠️ Phase 0 필수 확인**: 쿠팡 주문목록의 실제 DOM 구조는 로그인 상태에서만 확인 가능. **코딩 첫 작업은 반드시 실제 페이지에서 셀렉터 확정**(Claude Code로 개발자도구/스냅샷 분석). 이 문서는 구조를 가정하지 않는다.

### 4.5 AI 분류 설계 (Phase 3)
- 클라이언트 → Cloudflare Worker → Gemini Flash (배치: 상품명 최대 100개/요청)
- Worker가 라이선스 키 검증 후에만 AI 호출 (무임승차 방지)
- 프롬프트: 상품명 배열 → 카테고리 배열 JSON 반환. 캐시: 동일 상품명은 Worker KV에 캐시 (원가 절감 핵심)
- 예상 원가: 유저당 월 ~10원 수준 (Flash 배치 + 캐시 기준) → 마진 문제없음
- 사용자가 분류를 수정하면 CategoryRule(priority 최상)로 저장 → 다음부터 AI 호출 없이 적용

### 4.6 코딩 규칙 (Claude Code에게)
- 파일당 200줄 이하, 함수당 30줄 이하 유지
- 파싱 로직은 순수 함수로 분리 (HTML 문자열 in → OrderItem[] out) → 저장된 HTML 픽스처로 유닛 테스트
- 테스트: Vitest. 파싱/분류/합산 로직은 테스트 필수, UI는 선택
- 에러는 삼키지 말 것: 모든 catch는 사용자 알림 or 로그 중 하나에 반드시 연결
- 커밋 단위: 기능 하나 = 커밋 하나. Phase 완료 시 태그 (v0.1, v0.2…)

## 5. 수익화 설계

### 5.1 결제 구조
- **라이선스 키 방식**: 결제 → 이메일로 키 발급 → 확장에 입력 → Worker가 검증. 자체 회원가입 없음.
- 결제 대행(Merchant of Record) 사용: **Lemon Squeezy 또는 Paddle** — 개인이 사업자·PG 계약 없이 시작 가능하고 세금 처리를 대행해줌. (한국 간편결제 미지원이 단점 → 초기엔 카드 결제만으로 검증, 매출이 나오면 사업자등록 + 토스페이먼츠 전환 검토)
- ⚠️ 확인 필요: 수익 발생 시 사업소득 신고 의무. 연 매출이 유의미해지면 사업자등록 (이건 좋은 문제다)

### 5.2 가격
- 월 3,900원 / 연 29,000원 (연간 = 38% 할인으로 연간 유도)
- 7일 무료 체험 (키 없이 Pro 기능 로컬 체험 → 만료 시 잠금)
- 기준: 인디 확장 전환율 통계 2~5% (주간 활성 유저 기준)

### 5.3 손익 시뮬레이션
| 설치 | 주간 활성(40%) | 전환(3%) | 월 매출 |
|---|---|---|---|
| 1,000 | 400 | 12명 | ~4.7만원 |
| 5,000 | 2,000 | 60명 | ~23만원 |
| 20,000 | 8,000 | 240명 | ~94만원 |
| 50,000 + 다중 플랫폼 | 20,000 | 600명 | ~230만원 |

→ 쿠팡 단독으로는 상한이 낮다. **Phase 4 다중 플랫폼 확장이 본게임**이고, 쿠팡은 교두보.

### Phase 5 — 종결 시나리오 (시작 전에 정해두는 것)
